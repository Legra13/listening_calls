# Спецификация логики и структуры: Qolio Аналитика ТВК

Документ описывает полную логику подсчёта баллов, структуру данных и устройство HTML-отчёта для переноса в новое приложение.

---

## 1. Источники данных

Отчёт использует **два Excel-файла** из папки проекта:

### Источник A: `Результаты сотрудников по критериям.xlsx`
- Листы 7–20 (лист 8 пропускается)
- Только листы, содержащие колонку `Результат по сделке`
- **Формат:** агрегированные групповые баллы (уже посчитаны по группам)
- **Колонки:** Оператор, Дата презентации, Ссылка на сделку, Результат по сделке, [10 групп], Итог, МЕСЯЦ
- Групповые баллы хранятся как доли (0.0–1.0) → умножаются на 100 при загрузке
- Из URL сделки (`/deal/details/123`) извлекается `deal_id`
- Результирует в **`mainData`** — используется в Tab 1 и Tab 3

### Источник B: `data/Выгрузка полная ОП из qolio...xlsx`
- Один лист, полная выгрузка из Qolio
- **Формат:** 77 колонок: 6 мета-полей + 35 критериев + 35 комментариев + Итог + Общий комментарий
- Значения критериев: 0 или 1 (двоичные)
- `Итог` — дробное (0.0–1.0), нормализуется в % при загрузке
- Результирует в **`criteriaData`** — используется в Tab 2 и Tab 3 режим «Критерии»
- Записи, уже присутствующие в Источнике A (по `deal_id`), не дублируются в `mainData`

### Обогащение стадиями сделок
- Стадии загружаются из `Результат по сделке` (Источник A) и маппируются через `deal_id`
- Затем применяются к Источнику B: `df['stage'] = df['deal_id'].map(deal_to_stage)`
- Два целевых значения стадии:
  - `"сделка успешна"` → победа (Won)
  - `"не смог продать"` → поражение (Lost)
  - Всё остальное → в работе, не участвует в корреляционных расчётах

## 3. Логика подсчёта итогового балла

### Балл критерия
Хранится как 0 или 1 в исходных данных. При загрузке умножается на 100:
```
criterion_score = value * 100  → 0% или 100%
```

### Балл группы (в mainData — Источник A)
Берётся напрямую из Excel (уже посчитан): дробное значение × 100.

### Балл группы (при расчёте из criteriaData)
```
group_score = mean(критерии группы) * 100
```
Среднее арифметическое по всем критериям группы (не взвешенное).

### Итоговый балл (total_score)
Берётся из колонки `Итог` в обоих источниках.
- Если значение ≤ 1 — умножается на 100.
- Формула Qolio: взвешенная сумма `Σ(criterion_value × criterion_weight)` по всем 34 критериям.
- Максимум: 100 баллов (= 100%).

### Балл в режиме «Баллы» (в тепловой карте)
```
points = (pct / 100) × weight
```
Где `weight` — вес критерия или группы из таблицы выше. Показывает вклад в итоговые 100 баллов.

## 4. SUMIF-логика (корреляция)

Воспроизводит Excel-формулы SUMIF/SUMIFS:

### Средний балл по стадии сделки
```
avg_won  = SUMIF(stage, "сделка успешна",  col) / COUNTIF(stage, "сделка успешна")
avg_lost = SUMIF(stage, "не смог продать", col) / COUNTIF(stage, "не смог продать")
delta    = avg_won − avg_lost
```
Знаменатель — **общее** число строк с данной стадией, не только непустых.

### Win Rate (влияние выполнения блока)
```
done    = сделки, где col > 0
not_done = сделки, где col == 0
WR_done    = won_in_done / len(done) * 100
WR_not_done = won_in_not_done / len(not_done) * 100
WR_impact  = WR_done − WR_not_done  (в процентных пунктах)
```

### Диапазоны уровней выполнения (Tab 3, таблица 2)
| Диапазон | Метка | Описание |
|----------|-------|----------|
| = 0 | 0% | Блок не выполнен |
| > 0 и ≤ 40 | 1–40% | Слабое выполнение |
| > 40 и ≤ 70 | 40–70% | Частичное выполнение |
| > 70 и ≤ 100 | 70–100% | Хорошее выполнение |

Win Rate в ячейке = доля выигранных сделок среди всех сделок с известным итогом в данном диапазоне.

## 5. Цветовая схема (тепловые карты)

Трёхзонный градиент на основе значения в %:

| Зона | Диапазон | Цвет |
|------|----------|------|
| Красная | 0–40% | rgb(239,68,68) → rgb(251,146,60) |
| Жёлтая | 40–60% | rgb(251,146,60) → rgb(250,204,21) |
| Зелёная | 60–100% | rgb(250,204,21) → rgb(34,197,94) |

JavaScript-функция `heatBg(v)`: линейная интерполяция внутри каждой зоны.
Цвет текста: `#1e293b` (тёмный) для всех заполненных ячеек, `#94a3b8` (серый) для null.

## 7. Три вкладки отчёта

### Tab 1 — Общие показатели (использует mainData)

**KPI-карточки:**
- Звонков — `data.length`
- Ср. балл — среднее `total_score` по отфильтрованным записям
- Сделок успешно — count(stage == "сделка успешна"), % от закрытых
- Не смог продать — count(stage == "не смог продать"), % от закрытых

**Тепловая карта сотрудников:**
- Строки = сотрудники, столбцы = группы или критерии
- Ячейка = среднее значение по всем записям сотрудника
- Последняя строка «Команда» = среднее по всем данным
- Переключатели: «Группы / Критерии» и «% / Баллы»
- В режиме «Критерии» — двойная граница между группами (`border-left: 3px solid #4f8ef7`)
- В режиме «Критерии» используется **criteriaData**, в режиме «Группы» — **mainData**

**Динамика по неделям:**
- Ключ недели: `YYYY-WNN` (ISO week)
- График Chart.js (line), ось Y: 0–100%
- Таблица: неделя / кол-во записей / средний балл

### Tab 2 — Корреляция (mainData для групп, criteriaData для критериев)

Таблица и горизонтальная гистограмма, сортировка по убыванию Δ.

**Колонки таблицы:**
- Группа ТВК / Критерий
- ✓ Успешна (avg_won %)
- ✗ Не продал (avg_lost %)
- Δ (Won−Lost) — с цветным баром: зелёный если ≥0, красный если <0
- Win Rate выполнено (col > 0)
- Win Rate не выполнено (col == 0)
- Влияние WR = WR_done − WR_not_done (в pp)

В режиме «Критерии» строки группируются по группам (заголовки-разделители).

### Tab 3 — Корреляция по сотрудникам (mainData для групп, criteriaData для критериев)

**Таблица 1: Won / Lost / Δ по сотрудникам**
- Строки = сотрудники + строка «Команда»
- Столбцы = группы или критерии
- Ячейка: `Won% / Lost% / Δ`
  - Won% — среднее по записям с stage = "сделка успешна"
  - Lost% — среднее по записям с stage = "не смог продать"
  - Δ = Won% − Lost%, цвет зелёный/красный
- В режиме «Критерии» — двухуровневый заголовок (группа → критерии)

**Таблица 2: Win Rate по уровням выполнения**
- Строки = 4 диапазона (0% / 1–40% / 40–70% / 70–100%)
- Столбцы = группы или критерии
- Ячейка: Win Rate % + n (кол-во сделок)
- Цвет ячеек — стандартный heatBg(win_rate)
- Читать: если Win Rate растёт слева направо — блок влияет на закрытие сделки

## 8. Фильтры

Все фильтры работают на стороне клиента (JavaScript), перефильтровывают `RAW.mainData` и `RAW.criteriaData`.

| Фильтр | Применяется к |
|--------|---------------|
| Сотрудники (multiselect) | mainData + criteriaData |
| Год (multiselect) | mainData + criteriaData |
| Неделя (multiselect, зависит от года) | mainData + criteriaData |
| Диапазон дат (date range) | mainData + criteriaData |

Неделя кодируется как `YYYY-WNN`, список недель обновляется при изменении выбранных годов.
Функция `applyFilters()` вызывается при любом изменении и перерисовывает все компоненты.
