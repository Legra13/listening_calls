## Этап 4 — Форма оценки (4–5 ч)

**Цель:** оценщик может заполнить и сохранить оценку

- [ ] Страница «Новая оценка» (`GET /evaluations/new`):
  - Выбор чек-листа
  - Ввод deal_id → HTMX подтягивает данные сделки (оператор, дата, стадия)
  - Если deal не найден — ручной ввод имени оператора и даты
- [ ] Форма чек-листа: блоки-секции, внутри каждой — критерии с радиокнопками Да/Нет/Н/П + textarea для комментария
- [ ] Расчёт итогового балла при сохранении (логика из `logic_summary.md` §3)
- [ ] `POST /evaluations` — сохранение оценки и всех `evaluation_items`
- [ ] Проверка дубликата: предупреждение если deal_id уже оценён по этому чек-листу
- [ ] Страница просмотра оценки (`GET /evaluations/{id}`)
- [ ] Список оценок (`GET /evaluations`) с фильтрами: оператор, чек-лист, диапазон дат
- [ ] Редактирование и удаление оценки

## Этап 5 — Аналитика (6–8 ч)

**Цель:** все 3 таба + динамика — воспроизвести логику HTML-отчёта

- [ ] Движок фильтрации: сотрудник (мультиселект), год, неделя, диапазон дат → применяется ко всем табам
- [ ] **Tab 1 — Общие показатели:**
  - KPI-карточки: кол-во звонков, ср. балл, % успешных (от закрытых), % провальных
  - Тепловая карта сотрудников (строки) × блоки/критерии (столбцы), переключатель %/Баллы
  - Последняя строка «Команда» = среднее по всем
  - Цветовой градиент `heatBg(v)` — из `logic_summary.md` §5
  - Динамика по неделям: Chart.js line chart + таблица (неделя / кол-во / ср. балл)
- [ ] **Tab 2 — Корреляция (по блокам и критериям):**
  - `avg_won`, `avg_lost`, `delta`, `WR_done`, `WR_not_done`, `WR_impact` — по логике §4
  - Переключатель Блоки/Критерии
  - Сортировка по Δ (убывание)
  - Цветной бар для Δ: зелёный ≥ 0, красный < 0
- [ ] **Tab 3 — Корреляция по сотрудникам:**
  - Таблица 1: строки = сотрудники, столбцы = блоки/критерии, ячейка = Won%/Lost%/Δ
  - Таблица 2: Win Rate по диапазонам (0% / 1–40% / 40–70% / 70–100%)
  - Переключатель Блоки/Критерии
- [ ] Фильтры работают на сервере (не клиентский JS), перезапрос через HTMX
