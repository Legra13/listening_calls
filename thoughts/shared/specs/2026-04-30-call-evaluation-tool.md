# Инструмент оценки звонков — Спецификация

## Краткое описание

Внутренний веб-инструмент на Python для оценки звонков сотрудников коммерческих блоков. Заменяет Qolio (причина: дорогая лицензия + неудобные отчёты) и добавляет интеграцию с Битрикс24 для автоматического подтягивания итога сделки. Аудитория — 1–5 оценщиков внутри компании.

---

## Проблема

- Qolio — платная лицензия, неудобные отчёты
- Нет интеграции с Битрикс24: данные о стадии сделки (успех/провал) приходится вносить вручную или через Excel-экспорт
- Ключевой аналитики нет: нельзя быстро посмотреть корреляцию между блоками чек-листа и итогом сделки

---

## Критерии успеха

- Оценщик может заполнить оценку звонка полностью внутри нового инструмента (без Qolio)
- При вводе ID сделки система автоматически подтягивает данные из Битрикс24
- Все 3 таба аналитики (общие показатели, корреляция, корреляция по сотрудникам) + динамика по неделям доступны в реальном времени
- Инструмент работает без Qolio-лицензии

---

## Пользователи

**Роль: Оценщик / Администратор (единая роль)**
- 1–5 человек
- Технический уровень: средний (работают в Битрикс, Excel, Qolio)
- Цели: быстро заполнить чек-лист после прослушки, видеть отчёты
- Могут создавать и редактировать чек-листы

---

## User Journey

1. Пользователь открывает инструмент в браузере, вводит логин/пароль
2. Выбирает «Новая оценка»
3. Выбирает нужный чек-лист из списка (2–3 на старте)
4. Вводит ID сделки из Битрикса
5. Система обращается к MySQL Битрикс24 и подтягивает: имя оператора, дату, текущую стадию сделки
6. Пользователь слушает звонок в Битриксе (в отдельной вкладке) и заполняет чек-лист: по каждому критерию — **Да / Нет / Н/П** + опциональный текстовый комментарий
7. Сохраняет оценку
8. Переходит в раздел «Отчёты» и видит обновлённую аналитику

**Обработка ошибок:**
- ID сделки не найден в Битриксе → сообщение с предложением ввести вручную
- Стадия сделки ещё не определена (в работе) → оценка сохраняется, в аналитике корреляции эта запись не участвует (только в общих показателях)
- Сделка уже оценена → предупреждение с возможностью открыть существующую оценку

## Функциональные требования

### Must Have (P0) — MVP

**Оценка звонков:**
- Создание новой оценки: выбор чек-листа + ввод ID сделки
- Автоматическое подтягивание данных сделки из Битрикс24 MySQL
- Форма чек-листа: блоки → критерии, оценка Да/Нет/Н/П + комментарий к каждому критерию
- Сохранение оценки в БД
- Список всех оценок с поиском и фильтрами (сотрудник, дата, чек-лист)
- Редактирование/удаление оценки

**Управление чек-листами:**
- Создание чек-листа: название, блоки (название + вес), критерии (текст + вес) в каждом блоке
- Редактирование и архивирование чек-листов
- Предустановленный чек-лист ТВК (10 блоков, 34 критерия, веса — из `logic_summary.md`)

**Аналитика (все 3 таба + динамика):**
- Tab 1 — Общие показатели: KPI-карточки (кол-во звонков, ср. балл, % успешных, % провальных), тепловая карта сотрудников по блокам/критериям, динамика по неделям (график + таблица)
- Tab 2 — Корреляция: Won/Lost средние + Δ + Win Rate влияние по блокам и критериям
- Tab 3 — Корреляция по сотрудникам: Won/Lost/Δ в разрезе сотрудников, Win Rate по диапазонам выполнения
- Фильтры: сотрудник, год, неделя, диапазон дат
- Переключатели в тепловой карте: Группы/Критерии, %/Баллы

**Авторизация:**
- Вход по логину и паролю
- Управление пользователями (создание/удаление/смена пароля)

### Should Have (P1) — после MVP

- Импорт исторических данных из Excel (Qolio-формат + результаты сотрудников)
- Экспорт отчёта в Excel/CSV
- Уведомление оператору о результате оценки (email или в Битрикс)
- Автоматическое обновление стадии сделки (фоновая задача — периодически перепроверять незакрытые сделки)

### Nice to Have (P2)

- Встроенный плеер записи звонка (если Битрикс отдаёт URL записи)
- Многоуровневые чек-листы (блоки → подблоки → критерии)
- Сравнение нескольких чек-листов в отчёте

## Техническая архитектура

### Стек

| Слой | Технология |
|------|-----------|
| Backend | **FastAPI** (Python 3.11+) |
| Frontend | **Jinja2 + HTMX + Chart.js** (серверный рендеринг, без SPA-фреймворка) |
| БД приложения | **SQLite** (для MVP; легко мигрировать на PostgreSQL) |
| Интеграция Битрикс | MySQL (`mysql80.hostland.ru:3306`, `host1850712`) |
| Авторизация | **bcrypt** пароли + cookie-сессии (без JWT) |
| Деплой | VPS, systemd + uvicorn, nginx как reverse proxy |

> **Почему FastAPI + Jinja2 вместо Streamlit:** форма с 34 чекбоксами и комментариями — это классическая HTML-форма. Streamlit плохо подходит для сложных форм ввода и не умеет разделять URL-роуты. FastAPI + шаблоны дают полный контроль над UX и при этом остаются в Python-экосистеме.

### Модель данных

```
users
  id, username, password_hash, created_at

checklists
  id, name, description, is_active, created_by, created_at

blocks
  id, checklist_id, name, display_name, weight, order_index

criteria
  id, block_id, text, weight, order_index

evaluations
  id, checklist_id, deal_id, operator_name, eval_date, week_num, week_year,
  month, stage, total_score, evaluator_id, created_at, comment

evaluation_items
  id, evaluation_id, criterion_id, value (yes/no/na), comment

deal_cache
  deal_id, operator_name, deal_date, stage, last_synced_at
```

### Логика расчёта баллов

Воспроизводит логику из `logic_summary.md`:
- Балл критерия: `Да` = 100, `Нет` = 0, `Н/П` = исключается из знаменателя
- Балл блока: среднее по критериям (не взвешенное) × 100
- Итоговый балл: взвешенная сумма `Σ(criterion_value × criterion_weight)` по всем критериям (где `Н/П` = 0)
- Корреляция (SUMIF-логика): `avg_won`, `avg_lost`, `delta`, `WR_impact` — по `logic_summary.md` §4

### Интеграция с Битрикс24

**Таблицы в MySQL Битрикс24:**
- `b24-entera-bitrix24-ru-deals` — данные сделки
- `b24-entera-bitrix24-ru-deal-stages` — стадии

**Маппинг стадий:**
- `"сделка успешна"` → Won
- `"не смог продать"` → Lost
- Остальное → In Progress (не участвует в корреляции)

**Стратегия кеширования:** при вводе deal_id → запрос в Битрикс → сохранить в `deal_cache`. Кеш можно обновить вручную или через P1-фоновую задачу.

### Безопасность

- Пароли хранятся как bcrypt-хеши (salt rounds ≥ 12)
- Cookie-сессии с HttpOnly + Secure флагами
- Доступ к MySQL Битрикс24 — только чтение (SELECT)
- Нет публичного API, только веб-интерфейс
- CSRF-защита на всех формах изменения данных

## Открытые вопросы для реализации

1. **Схема таблиц Битрикс MySQL** — нужно проверить реальные имена колонок в `b24-entera-bitrix24-ru-deals` (название оператора, дата, стадия)
2. **Доступность MySQL с VPS** — проверить, доступен ли `mysql80.hostland.ru:3306` извне (или app должен жить на том же хостинге)
3. **Цветовая схема тепловой карты** — воспроизвести `heatBg(v)` из `logic_summary.md` §5
